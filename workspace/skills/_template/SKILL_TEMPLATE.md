# SKILL.md 작성 규칙

> skill-creator, skill-editor가 SKILL.md를 생성/수정할 때 반드시 이 규칙을 따른다.
> 이 파일 자체는 스킬이 아닌 레퍼런스 문서이다.

---

## 1. 파일 구조

모든 SKILL.md는 아래 구조를 따른다:

```
---
(frontmatter: YAML 메타데이터)
---

(본문: 에이전트 지침)
```

---

## 2. frontmatter 필드 정의

### 필수 필드

| 필드 | 타입 | 설명 | 예시 |
|------|------|------|------|
| `name` | string | 스킬 고유 식별자. 영문 소문자, 하이픈만 사용. 폴더명과 일치해야 함 | `weather-alert` |
| `description` | string | 스킬 기능 한 줄 요약. 에이전트가 라우팅 판단 시 참조 | `매일 아침 지정 지역의 날씨를 조회하여 Telegram으로 알림` |
| `user-invocable` | boolean | true: 사용자가 슬래시 커맨드로 직접 호출 가능. false: 모델 판단으로만 실행 | `true` |

### 선택 필드

| 필드 | 타입 | 설명 | 기본값 |
|------|------|------|--------|
| `disable-model-invocation` | boolean | true면 모델 프롬프트에서 제외 (비활성화 상태) | `false` |
| `metadata.openclaw.requires.env` | string[] | 실행에 필요한 환경변수 목록 | `[]` |
| `metadata.openclaw.requires.bins` | string[] | PATH에 있어야 하는 바이너리 목록 | `[]` |
| `metadata.openclaw.requires.anyBins` | string[] | 하나 이상 있으면 되는 바이너리 목록 | `[]` |
| `metadata.openclaw.os` | string[] | 지원 OS 목록 | `["linux"]` |
| `metadata.openclaw.always` | boolean | true면 모든 적격성 검사 우회 | `false` |

### frontmatter 작성 예시

```yaml
---
name: weather-alert
description: 매일 아침 지정 지역의 날씨를 조회하여 Telegram으로 알림
user-invocable: true
metadata:
  openclaw:
    requires:
      env: ["OPENWEATHER_API_KEY"]
      bins: ["python3", "curl"]
---
```

## 3. 본문 작성 규칙

### 3-1. 작성 형식 (에이전트 최적화)

```
DO:
  - 코드 블록으로 구조화 (Step, DO/DON'T, MUST/NEVER 등)
  - 테이블로 매핑 관계 표현
  - 게이트/체크리스트로 조건 분기 표현
  - 짧고 명확한 명령형 문장
  - 한 섹션이 한 주제만 다룸

DON'T:
  - 서술형 문장으로 긴 설명
  - 같은 내용 중복
  - "~할 수 있습니다", "~하면 좋겠습니다" 등 모호한 표현
  - 이모지
```

### 3-2. 필수 섹션

모든 SKILL.md 본문에는 아래 섹션이 반드시 포함되어야 한다:

```
1. 스킬 제목 (# 레벨)
   - > 트리거 조건 (어떤 요청이 이 스킬을 활성화하는지)

2. 동작 정의
   - 이 스킬이 구체적으로 무엇을 하는지
   - 입력 → 처리 → 출력 흐름

3. 플로우 (Step 1, 2, 3...)
   - 실행 순서를 단계별로 명시
   - 각 단계에서 에이전트가 해야 할 행동, 사용자에게 보여줄 내용, 대기 조건

4. 제약 (MUST/NEVER)
   - 반드시 지켜야 할 규칙
   - 절대 하면 안 되는 행동
```

### 3-3. 선택 섹션 (필요 시)

```
- 게이트: 조건 분기가 있을 때 (□ 조건? → Yes/No)
- 에러 처리: 실패 시 대응 방법
- 환경변수 안내: .env 수정이 필요한 경우 스텝 바이 스텝 안내 포함
- gateway 재시작: 재시작이 필요한 경우 AGENTS.md의 재시작 프로토콜 참조 명시
- 참조 파일: 스킬 폴더 내 보충 파일(스크립트 등) 설명
```

## 4. 보충 파일 규칙

SKILL.md 하나로 부족한 경우, 같은 스킬 폴더에 보충 파일을 둔다.

```
skills/weather-alert/
├── SKILL.md          ← 필수: 에이전트 지침
├── fetch_weather.py  ← 선택: 실행 스크립트
├── config.json       ← 선택: 설정 파일
└── README.md         ← 선택: 사용자용 설명 (에이전트는 SKILL.md만 참조)
```

### 보충 파일 참조 방법

SKILL.md 본문에서 `{baseDir}`로 스킬 폴더 경로를 참조한다:

```
실행: python3 {baseDir}/fetch_weather.py --city seoul
설정: {baseDir}/config.json
```

## 5. 스킬 유형별 가이드

### 5-1. 주기적 실행 스킬 (cron 트리거)

```
필수 명시 항목:
  - 실행 주기 (cron 표현식 또는 자연어)
  - 실행 시 수행할 작업 (구체적 명령/API 호출)
  - 출력 형식 (사용자에게 보여줄 메시지 템플릿)
  - 실패 시 동작 (재시도? 에러 알림?)
```

### 5-2. 요청형 스킬 (사용자 호출)

```
필수 명시 항목:
  - 트리거 키워드/패턴
  - 사용자 입력 파싱 규칙
  - 응답 형식
  - 대화 플로우 (질문-응답 순서)
```

### 5-3. 시스템 스킬 (내부 관리용)

```
필수 명시 항목:
  - 수정 대상 파일 매핑 (어떤 요청 → 어떤 파일)
  - 파일 수정 범위 (특정 섹션만? 전체?)
  - 수정 전후 표시 규칙
  - 승인 필요 여부
```

## 6. 글자수 제한

```
SKILL.md 상한: 15,000자
  - 75% 경고: 11,250자 이상이면 내용 압축 또는 보충 파일 분리 필요
  - 확인 방법: wc -m skills/[스킬명]/SKILL.md

초과 시 대응:
  1. 상세 설명이나 예시 → 보충 파일(README.md 등)로 분리
  2. 실행 로직 → 스크립트 파일로 분리, SKILL.md에서 {baseDir}/스크립트명 참조
  3. 그래도 초과 → 스킬을 분할 (하나의 큰 스킬 → 두 개의 작은 스킬)

전체 workspace 글자수 예산:
  - TOOLS.md "파일 글자수 제한" 참조
  - 스킬 추가 시 전체 합산 100,000자 이내 목표
  - 사용자 스킬이 많아져서 합산 초과 우려 시 → 사용자에게 안내
```

## 7. 작성 품질 체크리스트

스킬 생성/수정 완료 시 아래 항목을 확인한다:

```
□ frontmatter name이 폴더명과 일치하는가?
□ description이 라우팅 판단에 충분히 구체적인가?
□ 트리거 조건이 명시되어 있는가?
□ 플로우가 단계별로 구조화되어 있는가?
□ 각 단계에서 에이전트의 행동이 명확한가?
□ 제약 (MUST/NEVER)이 있는가?
□ 환경변수 필요 시 requires.env에 명시했는가?
□ 보충 파일 있으면 {baseDir} 경로로 참조했는가?
□ 서술형 문장 없이 구조화된 형식인가?
□ 15,000자 이내인가? (TOOLS.md 글자수 제한 참조)
□ 75% 경고선(11,250자) 미만인가? (초과 시 분리 검토)
```
