# 스킬 생성

> 트리거: "~만들어줘", "기능 추가", "새 스킬", "~하는 기능 있으면 좋겠다"

---

## 사전 참조

```
SKILL.md 작성 시 반드시 참조:
  skills/_template/SKILL_TEMPLATE.md
    - frontmatter 필드 정의
    - 본문 작성 규칙 (에이전트 최적화 형식)
    - 필수/선택 섹션
    - 품질 체크리스트
```

---

## 모드 판정 (요청 분석 후 자동 선택)

```
사용자 요청을 분석하여 간단 모드 / 상세 모드를 자동 선택한다.

간단 모드 조건 (아래 모두 충족 시):
  □ 외부 API 연동이 불필요 (환경변수 추가 없음)
  □ 추가 패키지 설치 불필요
  □ 요청 내용이 명확하여 추가 질문이 거의 불필요
  예: "매일 아침 오늘의 명언 알려줘", "인사하면 응원 메시지 보내줘"

상세 모드 조건 (아래 하나라도 해당 시):
  □ 외부 API 연동 필요 (환경변수 추가)
  □ 추가 패키지 설치 필요
  □ 요청이 복잡하거나 세부 사항 확인이 필요
  예: "매일 아침 서울 날씨 알려줘", "이메일 요약 기능 추가"

모드 안내:
  간단 모드 → "간단한 스킬이라 빠르게 만들겠습니다."
  상세 모드 → "몇 가지 확인이 필요합니다."
```

---

## 간단 모드 (3단계)

### Step S1: 스킬 이름 + 핵심 정보 확인

```
사용자 요청에서 이름, 트리거, 출력 형식을 자동 추론하고 한 번에 확인한다.

메시지 형식:
  "간단한 스킬이라 빠르게 만들겠습니다."
  ""
  "| 항목 | 값 |"
  "|------|-----|"
  "| 이름 | [추론된 이름] |"
  "| 트리거 | [추론된 조건] |"
  "| 출력 | [추론된 형식] |"
  "| 모델 | [기본 모델] |"
  ""
  "이대로 생성할까요? 수정할 부분이 있으면 말해주세요."
```

### Step S2: 공식 문서 조회 + SKILL.md 생성

```
승인 후:
  1. 관련 공식 문서 조회 (상세 모드 Step 7과 동일)
  2. SKILL_TEMPLATE.md 규칙에 따라 SKILL.md 생성
  3. 품질 체크리스트 확인
```

### Step S3: 완료 + 요약

```
상세 모드 최종 단계(Step 12)와 동일한 형식으로 요약 표시.
memory/에 생성 내역 기록.
```

---

## 상세 모드 (12단계, 한 단계씩 진행. 다음 질문 전 사용자 응답 대기)

### Step 1: 스킬 이름 확인

사용자 요청에서 스킬의 목적을 파악하고 이름을 제안한다.

```
이름 생성 규칙:
  - 영문 소문자 + 하이픈 (예: weather-alert, email-summary)
  - 목적이 드러나는 2~3단어

메시지 형식:
  "스킬 이름을 '[제안 이름]'으로 할까요?"
  "다른 이름을 원하면 말해주세요."
```

### Step 2: 트리거 조건

```
메시지:
  "언제 실행할까요?"
  "- 매일 특정 시간 (예: 매일 09:00)"
  "- 특정 주기 (예: 매시간, 매주 월요일)"
  "- 요청할 때만 (대화에서 직접 호출)"
```

### Step 3: 필요 정보 파악

```
메시지:
  "이 스킬을 실행하는 데 필요한 정보가 있나요?"
  "예: 위치 정보, 외부 서비스 API 키, 특정 URL 등"
  ""
  "없으면 '없어'라고 해주세요."

파악할 항목:
  - 외부 API 필요 여부 → 필요 시 API 이름, 환경변수명 결정
  - 추가 패키지 필요 여부 → 필요 시 패키지명 결정
  - 사용자 입력 필요 여부 → 필요 시 입력 형식 결정
```

### Step 4: 출력 형태

```
메시지:
  "결과를 어떤 형식으로 보여줄까요?"
  "- 간단 요약 (한두 줄)"
  "- 상세 보고 (항목별 정리)"
  "- 목록 형태"
```

### Step 5: 실행 모델 선택

```
□ 등록된 모델이 2개 이상인가? (openclaw.json → models.providers 확인)
  → Yes:
    모델 목록을 표시하고 선택 요청:
      "이 스킬을 실행할 모델을 선택해주세요."
      ""
      "등록된 모델:"
      "- [1] [모델명] ([provider]) -- 기본"
      "- [2] [모델명] ([provider])"
      ""
      "번호 또는 이름으로 선택해주세요. 기본 모델로 하려면 '기본'이라고 해주세요."
    → 선택 결과를 스킬 정보에 포함
  → No (모델 1개):
    자동으로 기본 모델 배치. 별도 질문 없이 Step 6으로 이동.
```

### Step 6: 요약 + 승인

수집된 모든 정보를 정리하여 표시한다.

```
표시 형식:
  "스킬 정보 정리:"
  ""
  "| 항목 | 값 |"
  "|------|-----|"
  "| 이름 | [스킬명] |"
  "| 트리거 | [조건] |"
  "| 입력 | [필요 정보] |"
  "| 출력 | [형식] |"
  "| 실행 모델 | [모델명] ([provider]) |"
  ""
  "필요 사항:"
  "- 패키지: [목록] (없으면 '없음')"
  "- 환경변수: [목록] (없으면 '없음')"
  ""
  "이대로 생성할까요?"
```

### Step 7: 공식 문서 및 최신 정보 조회

승인 후, SKILL.md 작성 전에 관련 공식 문서를 실시간으로 조회한다.

```
조회 대상 (스킬 내용에 따라 해당하는 것만):

  OpenClaw 관련:
    - https://docs.openclaw.ai/tools/skills → 스킬 구조, frontmatter 최신 스펙
    - https://docs.openclaw.ai/tools/creating-skills → 스킬 생성 가이드
    - https://docs.openclaw.ai/automation/cron-vs-heartbeat → cron/heartbeat 설정 (주기적 실행 스킬인 경우)

  외부 API 연동인 경우:
    - 해당 서비스의 공식 API 문서를 웹 검색으로 찾아 조회
    - API 엔드포인트, 인증 방식, 요청/응답 형식 확인
    - 무료 티어 제한, rate limit 등 실사용에 필요한 정보 확인

  패키지 설치가 필요한 경우:
    - 해당 패키지의 공식 문서에서 설치 방법, 호환성 확인

조회 결과 활용:
  - SKILL.md 작성 시 조회한 정보 기반으로 정확한 설정/명령어 사용
  - 스크립트 작성 시 조회한 API 스펙 기반으로 코드 작성
  - 조회한 출처를 SKILL.md 상단에 주석으로 기록

조회 실패 시:
  - "공식 문서를 확인하지 못했습니다. 학습 데이터 기준으로 작성하되, 실제 동작 확인이 필요합니다."
  - SKILL.md에 "# 주의: 공식 문서 미확인. 동작 검증 필요" 표기
```

### Step 8: SKILL.md + 보충 파일 생성

승인 후 실행:

```
생성 순서:
  1. skills/[스킬명]/ 폴더 생성
  2. skills/[스킬명]/SKILL.md 작성
    - SKILL_TEMPLATE.md 규칙에 따라 작성
    - frontmatter: name, description, user-invocable, metadata.requires
    - 본문: 에이전트 최적화 형식 (Step, DO/DON'T, MUST/NEVER 등)
  3. 실행 스크립트가 필요한 경우 → skills/[스킬명]/ 에 함께 생성
    - 스크립트 상단에 참고한 API 문서 URL 주석 추가
    - SKILL.md에서 {baseDir}/스크립트명 으로 참조

품질 확인 (SKILL_TEMPLATE.md 체크리스트):
  □ frontmatter name이 폴더명과 일치하는가?
  □ description이 라우팅 판단에 충분히 구체적인가?
  □ 트리거 조건이 명시되어 있는가?
  □ 플로우가 단계별로 구조화되어 있는가?
  □ 제약 (MUST/NEVER)이 있는가?
  □ 환경변수 필요 시 requires.env에 명시했는가?
  □ 15,000자 이내인가? (초과 시 보충 파일로 분리)

글자수 초과 시:
  → 상세 설명/예시를 보충 파일로 분리
  → SKILL.md에서 {baseDir}/파일명 참조
```

### Step 9: 패키지 설치 (필요 시)

```
조건: Step 3에서 추가 패키지가 필요한 것으로 파악됨

메시지:
  "스킬 실행에 필요한 패키지를 설치합니다."
  "설치할 패키지: [패키지명]"
  ""
  "진행할까요?"

승인 후:
  - 설치 명령 실행 (pip install, npm install 등)
  - 설치 결과 확인
  - 성공 → "패키지 설치 완료."
  - 실패 → 에러 메시지 표시 + 원인 분석 + 대안 제시
```

### Step 10: 환경변수 등록 안내 (필요 시)

```
조건: Step 3에서 환경변수가 필요한 것으로 파악됨

AGENTS.md의 "환경변수 변경 안내 프로토콜" 을 따른다:
  Step A → Step B → Step C (대기) → Step D (확인) → Step E (요약)

확인 성공 후 → gateway 재시작 필요 여부 판단:
  □ .env 파일이 변경되었는가?
    → Yes: AGENTS.md의 "gateway 재시작 전 기록 프로토콜" 실행
    → No: Step 9로 이동
```

### Step 11: 모델 배치 기록 (모델을 선택한 경우)

```
조건: Step 5에서 기본 모델이 아닌 다른 모델을 선택한 경우

AGENTS.md → "## 모델 배치" 섹션에 추가:
  "- [스킬명] → [provider/모델명]"

기존 "(없음)" 텍스트가 있으면 제거
```

### Step 12: 완료 + 전체 요약

```
표시 형식:
  "스킬 생성 완료."
  ""
  "[스킬명]"
  "- 위치: skills/[스킬명]/"
  "- 트리거: [조건]"
  "- 입력: [정보]"
  "- 출력: [형식]"
  "- 환경변수: [목록] (등록됨/미등록)"
  "- 상태: 활성"
  ""
  "수정이 필요하면 '[스킬명] 수정해줘'라고 말해주세요."

기록:
  1. memory/YYYY-MM-DD-[스킬명].md 생성
    - 스킬 생성 일시
    - 수집된 파라미터 전체
    - 설치된 패키지
    - 등록된 환경변수 (키 이름만, 값 제외)
    - 생성 과정에서 발생한 문제/해결 이력
  2. MEMORY.md 인덱스 추가
    - "[스킬명] 스킬 생성 완료 → memory/YYYY-MM-DD-[스킬명].md"
```

---

## 제약

```
MUST:
  - 한 단계씩 진행 (한 메시지에 여러 질문 금지)
  - 사용자가 건너뛰는 항목은 합리적 기본값을 제안하고 승인받기
  - SKILL.md 작성 시 skills/_template/SKILL_TEMPLATE.md 규칙 준수
  - 생성 결과를 memory/에 기록
  - 패키지 설치, 환경변수 등록은 반드시 사용자 승인 후 진행

NEVER:
  - API 키 값을 대화에 표시
  - 사용자 승인 없이 패키지 설치 또는 파일 생성
  - 서술형 문장으로 SKILL.md 작성 (에이전트 최적화 형식 필수)
  - 환경변수를 openclaw.json에 직접 추가 안내 (.env + reload.sh만 허용)
```
