# 규칙 편집

> 트리거: "~하지마", "항상 ~해줘", "밤에 알림 보내지마", "매일 ~확인해줘", "규칙 변경"

|-----------|-----------|-----------|

## 플로우

### Step 1: 규칙 종류/내용 파악

사용자 요청에서 규칙의 종류(행동/알림/주기적)와 구체적 내용을 추출한다.

```
파악할 항목:
  - 규칙 종류: 행동 규칙 / 알림 규칙 / 주기적 체크
  - 동작: 추가 / 수정 / 삭제
  - 규칙 내용: 구체적 조건과 행동 (예: "23:00~07:00 알림 금지")
```

### Step 2: 현재 규칙 + 변경 사항 표시

해당 파일을 실제로 읽어서 현재 규칙을 확인하고, 변경 사항을 표시한다.

```
추가의 경우:
  "규칙을 추가합니다."
  ""
  "추가할 규칙:"
  "- [규칙 내용]"
  ""
  "현재 사용자 규칙:"
  "- [기존 규칙 1]"
  "- [기존 규칙 2]"
  "(없으면 '없음')"
  ""
  "적용할까요?"

수정의 경우:
  "규칙을 수정합니다."
  ""
  "| 항목 | 현재 | 변경 후 |"
  "|------|------|---------|"
  "| [규칙] | [현재 내용] | [변경 내용] |"
  ""
  "적용할까요?"

삭제의 경우:
  "이 규칙을 삭제할까요?"
  "- [삭제 대상 규칙]"
```

### Step 3: 승인 → 파일 수정

```
AGENTS.md 수정 시:
  - "## 사용자 규칙" 섹션을 찾는다
  - 섹션이 없으면 AGENTS.md 하단 "## 사용자 규칙" 섹션 내에 규칙 추가
  - 기존 "(없음)" 텍스트가 있으면 제거하고 규칙 추가
  - 규칙 형식: "- [규칙 내용]"

HEARTBEAT.md 수정 시:
  - "## 사용자 추가 항목" 섹션을 찾는다
  - 기존 "(없음)" 텍스트가 있으면 제거하고 체크 항목 추가
  - 체크 항목 형식: "- [ ] [주기] [체크 내용]"
  - 예: "- [ ] 매일 09:00 읽지 않은 이메일 확인하여 요약 전송"
```

### Step 4: 변경 후 전체 규칙 목록 표시

```
"적용 완료."
""
"[현재 사용자 규칙]"
"- [규칙 1]"
"- [규칙 2]"
""
"[현재 주기적 체크 항목]"
"- [항목 1]"
"- [항목 2]"
```

### Step 5: 기록

```
기록 대상: memory/rules.md
기록 내용:
  - 변경 일시
  - 변경 유형 (추가/수정/삭제)
  - 변경 내용 (이전값 → 새 값)
  - 변경 요청 원문

MEMORY.md 인덱스:
  - 기존 "행동 규칙" 항목이 있으면 업데이트
  - 없으면 추가: "행동 규칙 변경 이력 → memory/rules.md"
```

## 레드라인 충돌 게이트

```
□ 사용자가 요청한 규칙이 레드라인과 충돌하는가?
  예: "API 키를 대화에 보여줘" → SOUL.md 레드라인 "민감 정보 대화 노출 금지"와 충돌

  → 충돌 시:
    "이 규칙은 보안 정책과 충돌합니다."
    "[충돌하는 정책 설명]"
    "이 규칙은 적용할 수 없습니다. 다른 방법을 제안해드릴까요?"
  → 비충돌: 정상 진행
```

## 제약

```
MUST:
  - AGENTS.md의 "사용자 규칙" 섹션만 관리 (다른 섹션 수정 금지)
  - HEARTBEAT.md의 "사용자 추가 항목" 섹션만 관리
  - 변경 전후를 함께 표시
  - 삭제 시 사용자 확인
  - 레드라인 충돌 검사 수행
  - 변경 결과를 memory/rules.md에 기록
  - 규칙 추가 후 글자수 확인:
    AGENTS.md 15,000자 상한 / HEARTBEAT.md 10,000자 상한
    75% 경고선 초과 시 → 사용자에게 "규칙이 많아지고 있어 정리가 필요합니다" 안내

NEVER:
  - AGENTS.md의 기본 동작 규칙, 스킬 라우팅, 메모리 운영, 레드라인 섹션 수정
  - HEARTBEAT.md의 기본 체크 항목 섹션 수정
  - 레드라인과 충돌하는 규칙 적용
```
