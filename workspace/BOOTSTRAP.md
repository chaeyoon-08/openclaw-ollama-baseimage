# BOOTSTRAP.md -- 최초 실행 초기화 리추얼

> 이 파일은 첫 세션에서 1회 실행된 후 자동 삭제된다.
> 삭제 후에는 온보딩 스킬이 후속 설정을 담당한다.

---

## 초기화 순서

에이전트가 첫 세션에서 사용자에게 응답하기 전에, 아래 점검을 자동으로 수행한다.
점검 결과를 사용자에게 보고한 후 온보딩으로 넘어간다.

### Step 1: 시스템 상태 점검

```
점검 항목:

□ Ollama 서비스 동작 확인
  방법: curl -sf http://localhost:11434/ 응답 확인
  성공: "Ollama: 정상"
  실패: "Ollama: 응답 없음 — 컨테이너 로그를 확인해주세요"

□ Ollama 모델 로드 확인
  방법: curl -sf http://localhost:11434/api/tags → models 목록에서 OLLAMA_MODEL 존재 확인
  성공: "모델: [모델명] 로드됨"
  실패: "모델: [모델명] 미로드 — 모델 다운로드가 완료되지 않았을 수 있습니다"

□ 웹 검색 도구 확인
  방법: openclaw.json → tools.web.search.enabled 값 확인
  성공: "웹 검색: 활성 (DuckDuckGo)"
  실패: "웹 검색: 비활성 — 정보 검색 기능이 제한됩니다"

□ Telegram 연결 확인
  방법: 이 메시지가 도달했으면 연결 성공
  성공: "Telegram: 연결됨"

□ 클라우드 저장소 확인
  방법: $STORAGE_PATH 디렉터리 존재 + 쓰기 가능 확인
  성공: "클라우드 저장소: 마운트됨 ([경로])"
  실패: "클라우드 저장소: 미마운트 — 백업 기능 사용 불가"

□ 외부 AI provider 확인
  방법: openclaw.json → models.providers 에서 ollama 외 provider 존재 확인
  있음: "추가 모델: [provider 목록]"
  없음: "추가 모델: 없음 (Ollama만 사용)"
```

### Step 2: 점검 결과 보고

```
메시지 형식:
  "시스템 초기화 완료."
  ""
  "[시스템 점검 결과]"
  "- Ollama: 정상"
  "- 모델: qwen3.5:9b 로드됨"
  "- 웹 검색: 활성 (DuckDuckGo)"
  "- Telegram: 연결됨"
  "- 클라우드 저장소: 마운트됨 (/mnt/storage)"
  "- 추가 모델: 없음"
  ""
  (실패 항목이 있으면)
  "[주의]"
  "- [실패 항목]: [상태] — [조치 안내]"
  ""

실패 항목이 있어도 온보딩으로 넘어간다 (치명적 실패가 아닌 한).
치명적 실패: Ollama 서비스 미응답 → "Ollama가 동작하지 않아 기능이 제한됩니다. 컨테이너 로그를 확인해주세요."
```

### Step 3: 온보딩 연계

```
점검 보고 직후, 온보딩 스킬의 Step 1(인사 + 자기소개)로 자연스럽게 전환한다.
별도 사용자 입력을 기다리지 않고 바로 이어간다.
```

## 주의사항

```
- 이 파일은 실행 후 OpenClaw가 자동 삭제함
- 삭제 후에도 온보딩 스킬은 정상 동작 (BOOTSTRAP.md와 독립)
- 시스템 점검에서 실패한 항목은 memory/에 기록하여 추후 참조 가능하게 함
- 점검 결과를 memory/bootstrap-YYYY-MM-DD.md에 기록
```
